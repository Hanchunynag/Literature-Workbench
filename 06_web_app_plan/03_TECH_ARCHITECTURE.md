# 技术架构

## 推荐方案

先做“快速个人版”，后面再逐步升级。

### 方案 A: 快速个人版

- 前端: Next.js
- 后端: Next.js Route Handlers
- 数据库: Supabase Postgres
- 文件存储: Supabase Storage
- 鉴权: Supabase Auth
- 异步任务: Trigger.dev
- AI 处理: Python 提取服务 + LLM 总结

这个方案适合先把产品跑起来。

### 方案 B: 长期可扩展版

- 前端: Next.js
- API 后端: FastAPI
- 数据库: PostgreSQL
- 文件存储: S3
- 异步任务: Celery + Redis
- 鉴权: 自建或 Supabase Auth
- AI 处理: 独立文献处理服务

这个方案适合后续扩成更稳定的研究平台。

## 我对你的建议

先做方案 A。

原因：

- 你现在最需要的是尽快用起来
- 单用户场景没必要一开始就拆太重
- 以后真的复杂了，再把提取和队列迁到独立后端即可

## 核心处理链路

### 上传即处理

处理流程建议固定为：

1. 用户上传 PDF
2. 文件保存到对象存储
3. 数据库创建一条 paper 记录，状态为 `uploaded`
4. 系统触发后台任务
5. 后台任务抽取文本和元数据
6. 执行重复检测
7. 若重复，则标记为 `duplicate`
8. 若非重复，则生成结构化摘要和分类建议
9. 写回数据库
10. 前端显示 `processed`

## 模块划分

### Web App

负责：

- 登录
- 上传
- 列表展示
- 详情展示
- 搜索筛选
- 手动修正

### Processing Worker

负责：

- PDF 文本提取
- 元数据标准化
- 重复检测
- 分类建议
- 提要生成
- 标签建议

### Storage Layer

负责：

- 原始 PDF
- 处理后的中间文件
- 缩略信息

### Database Layer

负责：

- 文献记录
- 分类体系
- 标签体系
- 处理日志
- 用户备注

## 与现有项目的衔接方式

你现在已有的本地流程不需要废掉，可以这样迁移：

- 把 `scripts/` 下提取逻辑封装成 API 可调用脚本
- 把 `metadata.json` 和 `clean_summary.md` 的结构转成数据库字段
- 保留本地文件作为原始研究资产
- 网站只做访问、处理、检索和编辑层

## 处理状态设计

建议给每篇论文加统一状态：

- `uploaded`
- `extracting`
- `deduplicating`
- `classifying`
- `processed`
- `duplicate`
- `needs_review`
- `failed`

## 重复检测建议

至少采用两层：

### 第一层

- 文件哈希
- 文件大小

### 第二层

- 标题归一化后相似度
- 作者首作者比对
- 年份比对
- DOI 比对

只靠文件名非常不稳，不建议继续作为主判据。

## 部署建议

如果你要手机和 PC 都能稳定访问，最省心的方式是：

- 前端部署到 Vercel
- 数据库和存储放 Supabase
- 域名绑你自己的域名
- 做成 PWA，手机上接近原生应用体验

## 为什么不建议继续靠固定时间自动处理

因为你的真实需求是“事件触发”：

- 我现在上传
- 系统现在处理
- 我现在就想看

这和“每天 21:00 统一扫描”是两个完全不同的产品逻辑。
